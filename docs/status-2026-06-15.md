## 二、待做

### P1 轨道 A 剩余 — `replay` 子命令（阻塞于数据）

**目标**：从「凭感觉调参」到「凭数据调参」的分水岭。新增 `trading_agent replay` 子命令。

| 能力 | 是否阻塞 | 说明 |
|---|---|---|
| paper 成交率（fill rate） | 否，本地可算 | 读历史 paper orders，统计成交 / 挂单 / 撤单比例 |
| blocked 原因分布 | 否，本地可算 | 聚合 decisions 日志里的 `blocked_reason`（no_trade_zone / chase_blocked / reward_risk_too_low / outside_entry_zone …）|
| score 桶 vs 1/3/5 日未来收益 | **是**，需多天数据 | 需要 yfinance 历史 + 多个 run date 的 `candidate_scores.json`；当前只有 1 个 run date |
| entry-zone 命中率 / breakout 成功率 | **是**，需多天数据 | 同上 |
| component attribution（各分量对收益的贡献） | **是**，需多天数据 | P2 校准排序权重的前置依赖 |

**建议**：先实现两个「本地可算」的（fill rate + blocked 分布），其余等积累 **2–3 周 paper runs** 后再做。本地部分可与 P2 并行，但 P2 真正的权重校准依赖 component attribution，所以 P2 只能先搭结构、不能定结论。

### P2 — 策略质量（price setup 纳入排序）

**现状**：排序发生在定价之前。`rank_candidates`（`policy/candidate_selector.py:117-137`）用：
```
trade_readiness_score =
    0.35 * candidate_total
  + 0.25 * technical_score
  + 0.10 * liquidity_score
  + 0.10 * research_score
  + 0.05 * catalyst_score        # 合计 0.85
```
而价格 setup 的质量信号——`reward_risk`、`risk_per_share`、setup_type（pullback/breakout）、是否在 entry zone——是在 `price_policy.py:68-71` **排序之后**才算出来的，**没能反哺排序**。结果：一个 reward:risk = 3.0、刚好在 entry zone 的票，和一个 reward:risk = 1.5、要追高的票，排序上无差别。

**P2 要做**：
1. 把 price setup 质量（`reward_risk_score` / `price_setup_score`）做成可计算的分量，纳入 `trade_readiness_score`。
2. 解决「排序在定价之前」的顺序问题——要么先轻量定价再排序，要么把 entry-zone 距离/RR 作为可提前估算的代理分量。
3. `selected[:20]` 等魔数配置化到 scoring_profile。

**关键约束（不可绕过）**：
> **新权重是起点不是结论。** 必须先用轨道 A 的 component attribution 校准再固化，否则只是把一组魔数换成另一组魔数。在没有 2–3 周回看数据前，P2 只能：(a) 搭好分量计算与接线，(b) 用**保守默认权重**并标注「待校准」，(c) 加测试锁住行为。**不要**在这个阶段宣称某组权重是最优。

**P2 涉及文件（预估）**：`policy/candidate_selector.py`（排序）、`policy/price_policy.py`（暴露 setup 质量）、`policy/loaders.py`（数据接线）、`src/config` 下的 scoring/policy profile、对应测试。

## 三、当前推荐的下一步

1. **先做 P2 的「结构 + 保守默认权重 + 测试」**，把 price setup 质量接进排序，但权重标注「待校准」。

3. **2–3 周后**回来用 replay 的 component attribution 校准 P2 权重 + 做 score 桶 vs 未来收益。

> 顺序提醒：P2 与「替换魔数」的诱惑相伴——守住「先搭结构、数据到位再定权重」这条线，是这一步最重要的纪律。
